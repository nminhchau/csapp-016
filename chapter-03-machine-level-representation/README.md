# Chapter 3: Machine-Level Representation of Programs

## Scope

This chapter explains how C programs are represented as x86-64 machine code. The main goal is not to write large programs in assembly, but to read compiler-generated assembly well enough to understand performance, debugging, procedure calls, data layout, control flow, and security vulnerabilities.

Machine code exposes details that C hides: registers, instruction addresses, condition codes, memory addressing, stack frames, and calling conventions.

## Key ideas

### 1. From C source to machine code

A C program is translated through several stages before it becomes an executable:

```text
C source -> preprocessor -> compiler -> assembler -> linker -> executable
```

Example command:

```bash
gcc -Og -o prog main.c mstore.c
```

Important options:

```text
-S    compile to assembly only
-c    compile and assemble to object code only
-Og   optimize while keeping code readable for debugging
-O1/-O2/-O3  stronger optimization levels
```

Example:

```bash
gcc -Og -S mstore.c      # produces mstore.s
gcc -Og -c mstore.c      # produces mstore.o
objdump -d mstore.o      # disassembles object code
```

### 2. Why learn assembly?

Reading assembly helps programmers:

- Understand what optimizations the compiler performs.
- Debug low-level program behavior.
- See how C constructs such as loops, conditionals, arrays, structures, and procedures map to machine operations.
- Understand buffer overflows, stack discipline, and control-flow attacks.
- Reason about performance-critical code.

Modern programmers usually read assembly generated by compilers rather than write large assembly programs by hand.

### 3. x86-64 overview

CS:APP focuses on x86-64, the 64-bit extension of the x86 family. x86 evolved from early 16-bit processors through IA32 and then to 64-bit x86-64.

Important historical consequences:

- The instruction set contains old compatibility features.
- Instructions have variable length, from 1 to 15 bytes.
- The architecture has a complex set of addressing modes.
- The current programming model is shaped by decades of backward compatibility.

### 4. Machine-level state

Machine code exposes processor state that C normally hides.

Key state in x86-64:

- Program counter `%rip`: address of the next instruction.
- Integer register file: 16 general-purpose 64-bit registers.
- Condition code registers: status bits from recent arithmetic or logical operations.
- Vector registers: hold integer or floating-point vector data.
- Memory: a byte-addressed virtual address space.

A machine instruction performs a simple operation such as copying data, adding values, reading memory, writing memory, or jumping to another instruction.

### 5. Registers

x86-64 has 16 general-purpose registers. Each 64-bit register has smaller named portions.

Common registers and conventional roles:

```text
%rax   return value
%rbx   callee-saved
%rcx   4th argument
%rdx   3rd argument
%rsi   2nd argument
%rdi   1st argument
%rbp   callee-saved / frame pointer in some code
%rsp   stack pointer
%r8    5th argument
%r9    6th argument
%r10   caller-saved
%r11   caller-saved
%r12   callee-saved
%r13   callee-saved
%r14   callee-saved
%r15   callee-saved
```

Register name suffixes indicate size:

```text
%rax   64-bit
%eax   low 32 bits
%ax    low 16 bits
%al    low 8 bits
```

Writing to a 32-bit register such as `%eax` clears the upper 32 bits of the corresponding 64-bit register.

### 6. Data formats and instruction suffixes

Many x86-64 instructions use suffixes to indicate operand size.

```text
Suffix   Meaning              Size
b        byte                 1 byte
w        word                 2 bytes
l        double word          4 bytes
q        quad word            8 bytes
s        single precision     4 bytes
d/l      double precision     8 bytes
```

Examples:

```asm
movb    # move byte
movw    # move word
movl    # move double word
movq    # move quad word
```

### 7. AT&T vs Intel assembly syntax

CS:APP uses AT&T syntax, which is common with GCC and `objdump` on Unix-like systems.

AT&T syntax characteristics:

- Register names begin with `%`, such as `%rax`.
- Immediate constants begin with `$`, such as `$5`.
- Source comes before destination: `movq %rax, %rbx` means copy `%rax` into `%rbx`.
- Instruction suffixes often indicate operand size: `movq`, `addl`, `cmpw`.

Intel syntax differs:

- Register names omit `%`.
- Immediate constants omit `$`.
- Destination often comes before source.
- Memory operands use a different notation.

Example contrast:

```asm
# AT&T
movq %rdx, %rbx

# Intel
mov rbx, rdx
```

### 8. Operand forms

Most instructions operate on operands. Operand types include immediate constants, registers, and memory references.

```text
Immediate: $Imm        constant value
Register:  %rax        value in register
Memory:    (%rax)      value stored at memory address in %rax
```

General memory addressing form:

```text
Imm(rb, ri, s)
```

Effective address:

```text
Imm + R[rb] + R[ri] * s
```

where `s` is usually 1, 2, 4, or 8.

Examples:

```asm
movq 8(%rbp), %rax          # load memory at %rbp + 8 into %rax
movq (%rax,%rdx,8), %rcx    # load memory at %rax + 8 * %rdx into %rcx
movq %rax, 16(%rsp)         # store %rax at memory address %rsp + 16
```

This addressing mode is useful for arrays, because indexing an array element often means computing:

```text
base + index * element_size
```

### 9. Data movement

The `mov` family copies data from a source to a destination.

```asm
movq $0x4050, %rax      # immediate to register
movw %bp, %sp           # register to register
movb (%rdi,%rcx), %al   # memory to register
movb $-17, (%esp)       # immediate to memory
movq %rax, -12(%rbp)    # register to memory
```

A single `mov` instruction cannot copy directly from one memory location to another. It must go through a register:

```asm
movq (%rdi), %rax
movq %rax, (%rsi)
```

### 10. Zero extension and sign extension

When moving a smaller value into a larger destination, the processor can either zero-extend or sign-extend it.

```asm
movzbq %al, %rax    # zero-extend byte to quad word
movsbq %al, %rax    # sign-extend byte to quad word
```

Zero extension fills high bits with 0. Sign extension fills high bits with the source sign bit.

Example:

```text
8-bit value: 0xFF
zero-extended to 64 bits: 0x00000000000000FF
sign-extended to 64 bits: 0xFFFFFFFFFFFFFFFF
```

### 11. Arithmetic and logical operations

Common integer operations include:

```asm
leaq    source, dest     # compute address expression, often used for arithmetic
incq    dest             # dest += 1
decq    dest             # dest -= 1
addq    src, dest        # dest += src
subq    src, dest        # dest -= src
imulq   src, dest        # dest *= src
xorq    src, dest        # dest ^= src
orq     src, dest        # dest |= src
andq    src, dest        # dest &= src
salq    k, dest          # left shift
sarq    k, dest          # arithmetic right shift
shrq    k, dest          # logical right shift
```

`leaq` computes an address expression but does not read memory. Compilers often use it for arithmetic.

Example:

```asm
leaq (%rdi,%rdi,4), %rax   # %rax = 5 * %rdi
leaq 7(%rax,%rdx,2), %rax  # %rax = %rax + 2 * %rdx + 7
```

### 12. Condition codes

Arithmetic and logical instructions set condition-code flags describing the result.

Important flags:

```text
CF   carry flag, for unsigned overflow
ZF   zero flag, result was zero
SF   sign flag, result was negative
OF   overflow flag, for signed overflow
```

Comparison instructions set flags without storing a result:

```asm
cmpq %rsi, %rdi    # compute %rdi - %rsi for flags
```

Test instructions set flags based on bitwise AND:

```asm
testq %rax, %rax   # checks whether %rax is zero or negative
```

### 13. Conditional moves and jumps

Control flow is implemented with jumps and condition codes.

Common conditional jumps:

```asm
je     jump if equal / zero
jne    jump if not equal / nonzero
jg     jump if greater, signed
jge    jump if greater or equal, signed
jl     jump if less, signed
jle    jump if less or equal, signed
ja     jump if above, unsigned
jb     jump if below, unsigned
```

Compilers may also use conditional moves to avoid branches:

```asm
cmovg %rdx, %rax   # if greater, copy %rdx to %rax
```

Conditional moves can improve performance when branch prediction would be unreliable, but they compute both candidate values before selecting one.

### 14. Loops and conditionals

C `if`, `while`, `for`, and `switch` constructs are translated into combinations of comparisons, jumps, and labels.

Example C loop:

```c
long sum(long *a, long n) {
    long result = 0;
    for (long i = 0; i < n; i++) {
        result += a[i];
    }
    return result;
}
```

Conceptual assembly structure:

```asm
movq $0, %rax        # result = 0
movq $0, %rdx        # i = 0
.Lloop:
cmpq %rsi, %rdx      # compare i and n
jge .Ldone
addq (%rdi,%rdx,8), %rax
addq $1, %rdx
jmp .Lloop
.Ldone:
ret
```

### 15. Switch statements and jump tables

A `switch` statement with dense case values may compile to a jump table. A jump table is an array of code addresses. The program computes an index, loads the target address, and jumps indirectly.

This can be faster than a long chain of comparisons when case values are compact.

Conceptual pattern:

```text
if index is in range:
    jump to jump_table[index]
else:
    jump to default
```

### 16. Procedures and the stack

Procedure calls require conventions for passing arguments, returning values, saving registers, and resuming execution after the call.

The stack grows toward lower addresses. `%rsp` points to the top of the stack.

Important instructions:

```asm
call target   # push return address and jump to target
ret           # pop return address and jump back
pushq src     # decrement %rsp and store src
popq dest     # load from %rsp and increment %rsp
```

On x86-64 System V systems, the first six integer or pointer arguments are passed in registers:

```text
1st: %rdi
2nd: %rsi
3rd: %rdx
4th: %rcx
5th: %r8
6th: %r9
```

The return value is placed in `%rax`.

### 17. Caller-saved and callee-saved registers

Registers are divided by convention.

Caller-saved registers may be overwritten by a called function. If the caller needs their values later, the caller must save them first.

Callee-saved registers must be preserved by the called function. If the callee uses one, it must save and restore it.

Examples:

```text
Caller-saved: %rax, %rdi, %rsi, %rdx, %rcx, %r8, %r9, %r10, %r11
Callee-saved: %rbx, %rbp, %r12, %r13, %r14, %r15
Stack pointer: %rsp must be restored before return
```

### 18. Arrays and pointer arithmetic

Machine code treats arrays as contiguous memory. For an array element `A[i]`, the address is:

```text
base_address + i * element_size
```

Example:

```c
long get(long *a, long i) {
    return a[i];
}
```

Possible assembly:

```asm
movq (%rdi,%rsi,8), %rax
ret
```

Here `%rdi` holds `a`, `%rsi` holds `i`, and `8` is the size of `long`.

### 19. Structures and alignment

Structures are laid out as contiguous bytes, with fields at fixed offsets. Compilers may insert padding to satisfy alignment requirements.

Example:

```c
struct rec {
    int i;
    long j;
    char c;
};
```

A 64-bit system may insert padding so that `j` starts at an address aligned to 8 bytes. This makes memory access more efficient but can increase structure size.

Programmers can reduce padding by ordering fields from largest alignment requirement to smallest when layout matters.

### 20. Buffer overflow and memory safety

Because C does not automatically check array bounds, writing past the end of an array can overwrite nearby stack data.

Dangerous example:

```c
void echo(void) {
    char buf[8];
    gets(buf);   // unsafe; never use gets
}
```

If input exceeds the buffer, it can overwrite saved registers or the return address. This is a classic buffer overflow vulnerability.

Safer code bounds the input:

```c
fgets(buf, sizeof buf, stdin);
```

Understanding stack layout explains why buffer overflows can hijack control flow.

### 21. Floating-point code

x86-64 uses different registers and instructions for floating-point values. Modern code commonly uses SSE/AVX registers rather than the older x87 floating-point stack.

Floating-point arguments and return values often use vector registers such as `%xmm0`, `%xmm1`, and so on.

Integer registers and floating-point registers follow different calling-convention rules.

## Worked example: from C to assembly

C source:

```c
long mult2(long, long);

void multstore(long x, long y, long *dest) {
    long t = mult2(x, y);
    *dest = t;
}
```

Representative x86-64 assembly:

```asm
multstore:
    pushq %rbx
    movq  %rdx, %rbx
    call  mult2
    movq  %rax, (%rbx)
    popq  %rbx
    ret
```

What happens:

1. `x`, `y`, and `dest` arrive in `%rdi`, `%rsi`, and `%rdx`.
2. `%rdx` is copied to `%rbx` because the call to `mult2` may overwrite `%rdx`.
3. `call mult2` computes the product and returns it in `%rax`.
4. `movq %rax, (%rbx)` stores the result at `*dest`.
5. `%rbx` is restored before returning because it is callee-saved.

## Useful commands

```bash
# Generate assembly
gcc -Og -S file.c

# Generate object file
gcc -Og -c file.c

# Disassemble object or executable
objdump -d file.o
objdump -d prog

# Show raw bytes in gdb
(gdb) x/14xb function_name
```

## Common pitfalls

1. Reading AT&T operands backward. In AT&T syntax, source comes before destination.
2. Forgetting that `leaq` computes an address expression without reading memory.
3. Confusing signed and unsigned conditional jumps.
4. Assuming every C variable has a memory location. Optimized code may keep values only in registers.
5. Assuming assembly output is identical across compiler versions or optimization levels.
6. Forgetting that a single instruction cannot move directly from memory to memory.
7. Ignoring calling conventions when reading procedure calls.
8. Overlooking structure padding and alignment.
9. Underestimating how dangerous unchecked buffer writes are in C.

## Summary

Chapter 3 connects C programs to their x86-64 machine-level representation. Assembly code reveals registers, memory addresses, condition codes, jumps, stack frames, procedure calls, data layout, and instruction encodings. Reading this representation helps programmers understand performance, debugging, compiler behavior, and security vulnerabilities that are hidden at the C source level.
